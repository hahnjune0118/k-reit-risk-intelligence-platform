# AX Requirements Definition

## 1. 목적과 범위

리츠 공시·세무검토의 반복 업무를 데이터 기반의 통제된 Workflow로 전환합니다. 범위는 공개자료 식별, 자산·과세정보 구조화, 법정 산식 계산, 검증상태 관리, 주요 세무쟁점과 요청자료, 검토메모 및 결과 출력입니다.

## 2. 주요 사용자

- 공개자료 기반으로 리츠 위험을 초기 검토하는 회계·세무 실무자
- 계산 근거, 검증 상태와 미해결 쟁점을 검토하는 책임자
- 데이터 계약, 규칙엔진과 출력 품질을 관리하는 운영 담당자

사용자는 기본적인 회계·세무 용어를 이해하지만 리츠 데이터 구조나 애플리케이션 구현 세부사항을 알지 않아도 핵심 결과를 확인할 수 있어야 합니다.

## 3. As-Is 업무분석

공시 PDF 검색 → 주소·소유관계 수작업 확인 → 공시가격 별도 조회 → Excel 세액 계산 → 근거 재확인 → 쟁점·요청자료·메모 수작업 작성

주요 문제는 자료 분산, 반복 확인, 계산과 근거의 분리, 담당자별 절차 차이, 공식자료 부족 시 추정값 사용 위험입니다.

## 4. 기능요건

| ID | 기능요건 | 검토 기준 |
|---|---|---|
| FR-01 | 공식자료 식별 | 출처, 기준일과 근거 상태를 기록한다. |
| FR-02 | 자산·필지·납세의무자 데이터 구조화 | 고유 식별자로 관계를 연결한다. |
| FR-03 | 세액 계산 | 검증된 Tax Rule Master와 Decimal 기반 산식을 사용한다. |
| FR-04 | 검증상태 표시 | 계산 완료, 일부 확인, 검토 필요와 데이터 부족을 구분한다. |
| FR-05 | 주요 세무쟁점 | 검증 결과에 따라 쟁점과 우선순위를 제시한다. |
| FR-06 | 추가 요청자료 | 각 쟁점에 필요한 증빙과 요청 사유를 연결한다. |
| FR-07 | 검토메모 | 계산, 근거, 쟁점, 한계와 후속조치를 구조화한다. |
| FR-08 | 결과 내려받기 | CSV·Excel·Markdown·HTML 형식을 제공한다. |

## 5. 비기능요건

| 항목 | 요건 |
|---|---|
| 재현성 | 같은 입력과 규칙 버전은 같은 계산 결과를 생성해야 한다. |
| 추적성 | 결과에서 Source Lineage와 적용 규칙을 확인할 수 있어야 한다. |
| 보안 | 인증정보와 비공개 값은 화면, 저장소와 출력물에 노출하지 않는다. |
| 데이터 정합성 | Schema와 참조키를 검증하고 중복·누락을 식별해야 한다. |
| Fail-closed | 필수 근거가 부족하면 임의 계산 완료 상태를 생성하지 않는다. |
| 사용자 가독성 | 위험, 근거, 요청자료와 검토 필요사항을 업무 순서에 맞게 표시한다. |

## 6. 데이터요건

- 자산, 필지, 건축물, 납세의무자와 Tax Rule Master를 분리 관리합니다.
- 출처 URL, 문서명, 기준일, Source Type과 검증상태를 보존합니다.
- 공시값, 공식자료, Snapshot, 추정값과 미확인 값을 구분합니다.
- 금액 계산은 부동소수점 오차를 피할 수 있는 Decimal 기반 처리를 유지합니다.
- 공개 Runtime은 검증된 CSV Snapshot과 서버 측 데이터 연결을 사용할 수 있어야 합니다.

## 7. 통제요건

- Source Grounding과 필수 컬럼 Schema 검증
- 자산·필지·납세의무자 참조 무결성 검증
- 입력 범위, 계산 상태와 Tax Rule 적용 가능성 점검
- 기존 Tax 계산값과 Assurance 진입점에 대한 회귀테스트
- 실제 고지서 대사 전 `verified_notice` 상태 생성 금지
- 최종 법적 판단과 신고·납부 의사결정에 대한 Human Review

## 8. 예외처리

- 필수 공식자료가 없으면 `data_insufficient` 또는 검토 필요 상태를 반환합니다.
- Source와 Schema가 검증되지 않으면 계산 또는 확정 상태로 진행하지 않습니다.
- 실제 고지액이 없으면 재계산액과 결정세액을 동일하게 표시하지 않습니다.
- API 연결이 제한되면 허용된 Snapshot으로 전환하되 Tax 필수 근거는 예시값으로 대체하지 않습니다.
- 출력 실패는 계산 결과를 변경하지 않고 사용자에게 재시도 가능한 상태로 알립니다.

## 9. 검수기준

1. 네 가지 공개 모드와 기존 Tax·Assurance 기능이 유지되어야 합니다.
2. AX 개요, Pain Point, As-Is·To-Be와 역할분담이 일반 화면 최상단에 표시되어야 합니다.
3. SK서린빌딩 Base 재계산 원시값이 기존 회귀 기준과 일치해야 합니다.
4. Source Grounding과 Fail-closed 문구 및 검증상태가 확인되어야 합니다.
5. 생성형 AI가 세법 판단을 수행하는 활성 Runtime으로 표현되지 않아야 합니다.
6. 버전, 문서 링크, 컴파일, 테스트, 정적검사와 패키지 정합성 검사가 통과해야 합니다.

## 10. 제외범위

- 신규 세법·계산 로직과 과세관청 결정세액 산출
- 신규 외부 API, 챗봇 또는 생성형 AI Runtime 연결
- 실제 신고·납부 의사결정과 최종 법률해석 자동화
- Deals, KRX 시장가치 분석 또는 신규 앱 모드
- 공식 Evidence가 부족한 다른 자산의 확정 계산
- 실측 근거가 없는 시간절감률, 정확도 개선률과 ROI 산정
